# Cognitive Loop Closure

## Goal

Connect the orchestration main loop so that action outcomes automatically trigger rumination, rumination candidates are auto-applied, and world model simulation feeds into branch evaluation — completing the end-to-end cognitive cycle.

## What I Already Know

- `AgentSearchOrchestrator.execute()` does retrieval→assembly→scoring→gating but stops there
- `RuminationService` has `schedule()` and `drain_short_cycle()`/`drain_long_cycle()` but they're never called from the orchestrator
- `WorldSimulator.simulate()` exists but is never called from the orchestrator
- `ValueAdjustment` candidates can be generated by rumination but never loaded back into `ValueScorer`
- Skill template candidates are generated but the "apply" step is separate

## Requirements

### 1. Rumination auto-trigger from orchestrator
- After `execute()` produces a `DecisionReport`, automatically schedule rumination trigger(s)
- Map gate decisions to rumination trigger types:
  - Warning → no rumination trigger (normal outcome)
  - SoftVeto → `MetacogVeto` trigger to SPQ
  - HardVeto/Escalate → `MetacogVeto` trigger to SPQ with higher priority
  - Action succeeded (Warning gate, no user correction) → `SessionBoundary` or `IdleWindow` to LPQ

### 2. Short-cycle rumination drain after assembly
- When rumination triggers are scheduled, drain SPQ immediately
- Apply short-cycle writebacks (LocalAdaptationEntry) to the next assembly request

### 3. World model simulation in branch evaluation
- Before value scoring, optionally simulate the top action candidate
- Feed simulation results into `ActionBranch` risk_markers and expected_effects
- This is opt-in via `AgentSearchRequest` flag

### 4. Value adjustment application from rumination
- When constructing `WorkingMemoryScoringPort`, check for persisted value adjustment candidates
- Apply them to base config using `from_persisted_adjustments()`

### 5. Full cycle method
- Add a `cycle()` method to the orchestrator that chains: execute → rumination trigger → drain → apply
- Returns both the decision report and any rumination writeback summary

## Acceptance Criteria

- [ ] `execute()` auto-triggers rumination based on gate decision
- [ ] SPQ drain happens synchronously after execute
- [ ] World model simulation can optionally enrich branch risk markers
- [ ] Value scorer loads persisted adjustments from rumination candidates
- [ ] `cycle()` method provides end-to-end: execute + rumination + apply
- [ ] All existing tests still pass
- [ ] New tests for auto-trigger, drain, simulation enrichment
- [ ] `cargo fmt --check`, `cargo clippy --tests -- -D warnings`, `cargo check --tests` pass

## Out of Scope

- Background scheduler for LPQ drain (still manual/test-driven)
- Long-cycle rumination auto-drain (only SPQ for now)
- Persistence of rumination state across sessions

## Technical Notes

- Primary files: `src/agent/orchestration.rs`, `src/cognition/rumination.rs`, `src/cognition/value.rs`
- The orchestrator needs a `RuminationService` reference (new dependency)
- Simulation is opt-in to avoid breaking existing behavior
