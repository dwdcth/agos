# Milestone v1.1: Embedding Second-Channel Retrieval

**Status:** ✅ SHIPPED 2026-04-17
**Phases:** 08-10
**Total Plans:** 8

## Overview

v1.1 在不破坏 lexical-first 基线和 explainability 的前提下，把 embedding 检索扩展成 ordinary retrieval 的 second channel。这个里程碑先补齐 embedding foundation，再把 lexical-only / embedding-only / hybrid 三种运行模式接进同一个 `SearchService` 合同，最后收敛 operator diagnostics 与 agent-search compatibility，保证新通道不会变成一条单独的 semantic bypass path。

## Phases

### Phase 08: Embedding Backend And Index Foundation

**Goal**: 为 optional embedding second-channel 建立最小可用底座，包括 backend config/readiness、embedding 持久化，以及 additive vector sidecar/index。  
**Depends on**: Phase 7  
**Plans**: 3 plans

Plans:

- [x] 08-01: Add embedding backend config, readiness, and storage contracts
- [x] 08-02: Persist embeddings for ingested records or chunks
- [x] 08-03: Add optional vector sidecar/index bootstrap and maintenance path

**Details:**

- 添加可选 embedding backend/model 配置，同时保持 lexical-only baseline 稳定。
- 将 chunk-aligned embeddings 以 additive 方式持久化，不替换 authority store。
- 把 vector sidecar/index readiness 暴露到 `status` / `inspect schema`，作为与 lexical readiness 分离的 substrate state。

### Phase 09: Dual-Channel Retrieval Fusion

**Goal**: 在 ordinary retrieval 中引入 lexical-first + embedding second-channel 的 recall / fusion / rerank，使 dual-channel 结果仍然保持 explainable。  
**Depends on**: Phase 8  
**Plans**: 3 plans

Plans:

- [x] 09-01: Add embedding second-channel recall on top of the existing search contract
- [x] 09-02: Implement lexical-first fusion, dedupe, and rerank logic
- [x] 09-03: Extend result traces to explain dual-channel contribution

**Details:**

- 从真实根级 `config.toml` 派生 lexical-only / embedding-only / hybrid 三种 retrieval mode。
- 让一个共享的 `SearchService` 支持 lexical recall、embedding recall 和 hybrid fusion。
- 在最终 `SearchResult` trace 中显式报告 lexical / embedding / hybrid contribution，同时保留 lexical citation/provenance 作为主解释表面。

### Phase 10: Dual-Channel Diagnostics And Service Compatibility

**Goal**: 把 dual-channel retrieval 的状态诊断、CLI/library surface 和 agent-search 兼容性补齐，确保新通道不会破坏现有 ordinary retrieval / agent-search 边界。  
**Depends on**: Phase 9  
**Plans**: 2 plans

Plans:

- [x] 10-01: Extend dual-channel diagnostics and operator surfaces
- [x] 10-02: Ensure ordinary retrieval and agent-search compatibility over the same dual-channel service

**Details:**

- 为 `search` 和 `agent-search` 增加统一的 `--mode lexical_only|embedding_only|hybrid` 控制面。
- 在 `status` / `doctor` 中增加 `active_channels` 与 `gated_channels`，让 operator 可以准确判断 lexical 和 embedding 的运行状态。
- 保持 agent-search 继续复用 ordinary retrieval seam，并用运行时完整 config 保留 embedding/vector 行为。

## Milestone Summary

**Decimal Phases:**

- None.

**Key Decisions:**

- 决定保持 lexical-first 作为稳定基线，embedding 只作为 second channel，而不是替代 lexical explanation。
- 决定从真实 repo-root `config.toml` 派生 dual-channel mode 变体，而不是维护单独测试配置宇宙。
- 决定让 agent-search 透传完整 runtime config 复用 ordinary retrieval，而不是新开 semantic-only bypass path。

**Issues Resolved:**

- embedding backend、vector sidecar/index readiness 不再只是 foundation substrate，而是成为 operator 可读的 status/doctor surface。
- dual-channel retrieval 的 final traces 明确区分 lexical-only / embedding-only / hybrid contribution。
- agent-search 在 dual-channel 模式下保留 ordinary retrieval 的 citation 和 trace 合同。

**Issues Deferred:**

- richer embedding lifecycle management、background re-embedding 和 index rebuild tooling。
- MCP / HTTP interface surface。
- richer human review / governance UX。

**Technical Debt Incurred:**

- No blocking tech debt recorded in the v1.1 audit.
- 环境外部仍会出现 `/home/tongyuan/.cargo/config` deprecation warning，但不影响仓库功能或验证结果。
