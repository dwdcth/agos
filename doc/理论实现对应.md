# 理论-实现对应审计

> 目的：校正 `doc/0415-00记忆认知架构.md` 与当前 Rust 代码库之间的对应关系。本文只记录“代码和测试能证实什么”，并显式区分：
>
> - `✅ 已实现`：默认或受支持运行路径上已接通，且有代码/测试证据
> - `⚠️ 部分实现`：数据结构、服务或测试存在，但不是默认路径，或只实现了子集
> - `❌ 未实现`：当前代码库中没有可运行实现

---

## 1. 总体结论

当前仓库已经实现了一个相当完整的本地优先记忆/检索底座，以及一套可运行的认知编排骨架，但旧版对应表有两类明显失真：

1. 把已经支持的能力写成“未实现”
2. 把仅存在于底层服务、测试或可选路径中的能力写成“默认已落地”

本次审计后的结论是：

- 词法优先检索、taxonomy 分类、DSL 摘要落盘、工作记忆装配、价值投影、元认知门控已实现；`embedding_only` / `hybrid` 的检索代码路径也已实现，但运行仍受 embedding backend、model、vector backend 与 sidecar/index 前置条件约束
- 反刍、世界模型快照、自我模型快照、技能模板 runtime read-model 等能力已实现为库层/服务层能力，但不能等同于“所有默认命令都完整闭环使用”
- 行动执行器、真正的世界状态演化接口、图关系层、完整的默认认知学习闭环，仍是部分实现或未实现

## 3. 关键纠偏

### 3.1 记忆分类并非“未实现”

旧说法：

- “未实现分类器，DSL 仍由 LLM 生成”

当前事实：

- `src/memory/classifier.rs` 已实现 `KeywordTaxonomyClassifier`
- `src/memory/pipeline.rs` 的 `DefaultMemoryPipeline::default_v1()` 使用 `KeywordTaxonomyClassifier + RuleBasedSummaryGenerator`
- `tests/memory_classifier.rs`、`tests/memory_pipeline_default.rs` 已覆盖分类输出
- `src/ingest/mod.rs` 的默认 ingest 路径优先走 rule-based summary；只有在配置了 LLM 且 summary backend 选择 Rig 时，才进入 Rig summary 路径

审计结论：

- `✅ taxonomy 分类器已实现`
- `✅ 默认 DSL 摘要并不依赖 LLM`
- `⚠️ Rig summary 是可选增强路径，不是唯一实现`

### 3.2 双通道检索并非“保留未实现”

旧说法常把 `embedding_only` / `hybrid` 说成“Phase 1 保留”。

当前事实：

- `src/search/mod.rs` 已实现 `RetrievalMode::LexicalOnly / EmbeddingOnly / Hybrid`
- `embedding_recall()`、候选合并、通道 trace、降级逻辑都已存在
- `tests/dual_channel_retrieval.rs`、`tests/retrieval_cli.rs`、`tests/agent_search.rs` 已覆盖双通道检索与 agent-search 复用
- 本次同步修正了 `src/core/app.rs`、`src/core/status.rs`、`src/core/doctor.rs` 中的 readiness/gating 语义，使其和真实运行前置条件一致，不再把缺失 `vector.backend` 的配置误报为 ready

审计结论：

- `✅ embedding_only / hybrid 检索代码已实现`
- `✅ agent-search 已复用普通检索模式`
- `⚠️ 是否可运行仍取决于 embedding backend、model、vector backend、vector sidecar/index 是否就绪`

### 3.3 认知闭环不能一概写成“默认已完成”

当前事实：

- `src/agent/orchestration.rs` 同时有 `execute()` 与 `cycle()`
- `cycle()` 才会构造 rumination trigger、调度队列、drain short-cycle
- 默认 CLI `agent-search` 路径当前仍以 `run()` / `execute()` 报告为主，通过 `RigAgentSearchAdapter` 暴露结果
- 反刍服务 `src/cognition/rumination.rs`、治理桥接、候选落盘、短周期写回都已经实现并有测试，但默认 CLI 并未把“所有一次 agent-search 都自动进入持久化学习闭环”作为已承诺行为

审计结论：

- `✅ 认知循环骨架已实现`
- `⚠️ 默认命令路径的完整学习闭环不是全自动已落地能力`

## 6. 默认运行路径 vs 可选能力

### 默认更可信的能力

- `ingest` 会把记录落库，并为每条记录生成 taxonomy + DSL sidecar
- `search` 会执行 lexical / embedding / hybrid 检索，并保留 citation / trace
- `agent-search` 会执行检索、工作记忆装配、价值投影、元认知门控，并输出结构化报告

### 需要条件或额外接线的能力

- `embedding_only` / `hybrid` 需要 builtin embedding backend、模型、`vector.backend=sqlite_vec`、vector sidecar/index 都就绪
- 自我模型/技能模板/世界模型的 runtime persisted read-model 依赖 `subject_ref` 和相关持久化状态
- rumination 写回与长周期候选依赖显式调用 `RuminationService` / `cycle()` 路径

---

## 7. 仍然存在的主要差距

| 差距 | 状态 | 备注 |
|---|---|---|
| 世界模型作为长期动态模型的使用深度不够 | ⚠️ | 现在更像可选快照/预测层，而不是默认主导层 |
| 默认 agent-search 仍偏“决策支持报告” | ⚠️ | 行动选择有了，但真实执行链薄 |
| 反刍不是所有默认命令的自动后处理 | ⚠️ | 服务具备，默认产品路径未完全承诺 |
| 图关系/跨主题关联层缺失 | ❌ | 仍以树状主路径为主 |
