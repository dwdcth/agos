# 0415 记忆认知

> 定位：这是 `agent_memos` 背后的认知理论总纲。  
> 它不是实现文档，不讨论表结构、接口和模块命名，而讨论一个问题：**什么是记忆，以及记忆在认知系统中处于什么位置。**

---

## 0. 导言：从“记住”与“回忆”开始

“记忆”这两个字本身就包含一个最基本的结构：

- 记住
- 回忆

如果只有“记住”，那只是存储。
如果只有“回忆”，那只是检索。
真正的记忆，必须同时回答：

1. 什么东西被保留下来？
2. 它是如何被组织起来的？
3. 它为什么能在未来重新进入前台？
4. 它如何随着新的经验被修正、沉淀、退场？

因此，本文不把记忆理解为“数据”，也不把记忆理解为“搜索结果”，而把它理解为：

> **认知系统为了服务未来行动，而对过去经验进行结构化保存、持续更新、状态化召回和前台调度的机制。**

这个定义有三个关键信号：

- 记忆服务的是未来，而不是过去。
- 记忆的本体是结构，而不是原文。
- 记忆既是状态，也是过程。

## 1. 记忆不是仓库，而是未来行动的准备系统

### 1.1 记忆不等于信息保留

一个日志系统能保留过去发生了什么，但它不一定具备记忆。

一个向量数据库能返回相似片段，但它不一定具备记忆。

一个知识库能组织事实，但它也不一定具备记忆。

因为真正的记忆不只是“有东西在”，而是：

> **有东西能在恰当的时候，以恰当的形式，再次进入前台。**

也就是说，记忆的核心不在于“保存”，而在于“可再次调用”。

### 1.2 记忆的本体是可再次调用的结构

原始经验是记忆的原料，但不是记忆的完成态。

一段对话、一条日志、一次失败、一次成功、一个配置文件，都只是经验材料。
只有当这些材料被重新编码为未来可用的结构时，记忆才真正形成。

因此可以把记忆写成一句简洁定义：

> **记忆，是把过去的信息重组为未来可再次调用的结构。**

### 1.3 遗忘不是删除，而是退出前台

如果记忆服务的是未来行动，那么遗忘就不应首先被定义为“数据没了”。

更合理的定义是：

> **遗忘，是某个结构不再优先进入前台。**

这意味着：

- 它可能还保留在后台；
- 它可能仍然有证据；
- 它可能在特定情境下重新被唤起；
- 它只是退出了默认的前台竞争。

所以，“记住 / 回忆 / 遗忘”并不是三个分离的动作，而是一个连续机制中的三种状态。

## 2. 认知系统与纯检索系统：两块系统，两个层次

### 2.1 纯检索系统

一个纯检索系统并不一定是中性的数据库。
在这里，它被重新定义为：

> **原生支持状态输入的 recall engine。**

它负责：

- 保存结构化经验；
- 提供可解释召回；
- 维护时间正确性；
- 返回证据与 trace；
- 接受情境、情绪调制、目标偏置作为召回参数；
- 但不让这些状态改写真值。

它回答的是：

> 在当前状态下，哪些有效结构最值得被召回？

### 2.2 加上 agent 的认知系统

一旦系统加入 agent，它就不再只是 recall engine，而会出现：

- 自我模型
- 目标
- 注意力
- 工作记忆
- 行动
- 元认知
- 价值层

于是系统的问题变成：

> 我现在该如何理解世界，并据此行动？

所以两块系统的关系可以写成：

- 纯检索系统解决 recall
- 认知系统解决 cognition

### 2.3 它们的边界

这个边界不能混。

如果把纯检索系统直接当成认知系统，系统会退化成“高级搜索”。
如果把认知逻辑全部压进底层检索，系统会失去稳定边界与通用性。

更稳的结构是：

> **检索系统负责“可以想起什么”，认知系统负责“现在该想起什么、怎么用、为什么这样做”。**

## 3. 九层认知架构

在本文中，认知系统由九层共同构成：

1. 记忆系统
2. 世界模型
3. 自我模型
4. 技能记忆层
5. 注意力系统
6. 工作记忆
7. 行动系统
8. 元认知层
9. 价值/偏好/规范层

它们之间的关系不是线性流水线，而是一个环路。

### 3.1 记忆系统

记忆系统负责把经验变成可追溯、可更新、可调用的结构。

它保留：

- 经验材料
- 证据链
- 时间痕迹
- 局部结论
- 可被重新组织的历史

它不直接决定行动，但它决定行动是否有可调用的过去。

#### 3.1.1 记忆首先应当是结构化记，而不是平铺记

如果“记”只是把原文按时间顺序堆起来，那么系统最多拥有资料，不拥有记忆。

更稳的做法是把记忆组织成一棵可演化的结构树，而不是一条平铺日志。

这里进一步拍板：

> **主分类采用固定层级，而不是无限层。**

推荐的主路径为：

- Domain
- Topic
- Aspect
- Kind
- Fact

这个层级不是为了模仿文件夹，而是为了给记忆系统一个稳定的“主挂载位置”。

它必须体现一个原则：

> **越往上越抽象，越往下越具体；越往上越稳定，越往下越贴近证据。**

也就是说，记忆系统不只是存“内容”，还要存“内容被放在什么认知位置上”。

它更像分类学中的“门、纲、目、科、属、种”：

- 上层负责主题边界与组织秩序；
- 中层负责语义分叉与任务可导航性；
- 底层负责真正可被引用、验证、修订和调用的事实节点。

因此，一个成熟的记忆系统不应把“主题”“分类”“事实”混成同一种对象。
它们都属于记忆，但它们承担的是不同粒度的认知职责。

#### 3.1.1.1 为什么不用无限层

无限层看起来灵活，但在认知系统里通常意味着：

- 同类事实会被挂到不同深度；
- LLM 很难稳定地产生一致层深；
- 人工维护会不断陷入“这一层要不要再细分一层”的争论；
- 检索裁剪和统计聚合都会变得不稳定。

所以，主结构应当先保证：

- 层数固定；
- 每层职责固定；
- 事实永远有唯一主路径。

更细的差异，不应继续往树上加层，而应交给：

- DSL 字段；
- 标签；
- 图关系。

#### 3.1.1.2 为什么主结构用树，而不是纯图

树更适合回答：

> **这条记忆主要属于哪里？**

图更适合回答：

> **这条记忆还和谁有关？**

因此更合理的组合是：

- **树负责主路径**
- **图负责跨主题关联**

也就是说，系统不应从一开始就把记忆主结构做成纯图。
纯图太自由，容易导致“什么都相关”，最后失去主组织秩序。

#### 3.1.1.3 四层主分类的职责

固定四层主分类时，每一层应承担明确认知职责：

##### 1. Domain

Domain 是最大认知域。

它回答：

> 这条记忆属于哪一类世界区域？

例如：

- `project`
- `product`
- `user`
- `team`
- `system`

Domain 不负责细节，它负责决定记忆首先进入哪个大世界。

##### 2. Topic

Topic 是域内的稳定主题。

它回答：

> 在这个认知域里，系统当前讨论的是哪一块稳定对象？

例如：

- `auth`
- `billing`
- `memory`
- `retrieval`
- `deployment`

Topic 是记忆系统的主导航轴之一。

##### 3. Aspect

Aspect 是对主题的观察切面。

它回答：

> 这条记忆是在谈主题的哪个方面？

例如：

- `provider`
- `cost`
- `policy`
- `latency`
- `timeline`

Aspect 不是事实类型，而是观察角度。

##### 4. Kind

Kind 是认知对象类型。

它回答：

> 这条东西究竟是什么性质的记忆？

例如：

- `decision`
- `fact`
- `risk`
- `constraint`
- `hypothesis`
- `pattern`

Kind 用来防止“供应商选择”和“风险提醒”被挂在同一种对象类型下。

##### 5. Fact

Fact 是叶节点。

它承载：

- DSL 摘要；
- 原文锚点；
- 真值层；
- 时间；
- 来源；
- 关联关系。

真正可被引用、比较、修订和装配进工作记忆的，是 Fact，而不是上面的抽象层。

#### 3.1.1.4 主路径约束

如果系统采用固定层级，那么应有以下约束：

1. **每条事实必须有唯一主路径**
   也就是必须能落到一个明确的 `Domain/Topic/Aspect/Kind` 下。

2. **主路径不能跳层**
   不能直接从 Domain 挂到 Kind，也不能只写 Topic 不写 Aspect。

3. **分类层应尽量稳定，事实层允许频繁增删**
   上层是认知秩序，下层才是动态内容。

4. **当分类不确定时，不新增层，而是使用保守占位**
   例如 `aspect=general`，而不是临时发明一层。

5. **跨主题联系不用树内重复挂载解决**
   用图关系解决，不用让一条事实同时出现在多个主路径上。

#### 3.1.1.5 命名规则

为了让 LLM 和程序都能稳定读写，主分类建议采用以下命名规则：

- 小写英文
- 多词使用 `snake_case` 或单词短语，不使用自然句子
- 每层都尽量是“名词或名词短语”，而不是完整判断句
- 不在主路径里写时间、情绪、来源

例如：

- 好：`project / auth / provider / decision`
- 好：`system / retrieval / latency / risk`
- 不好：`project / 我们四月改了认证 / 很重要 / 选择了 Clerk`

#### 3.1.1.6 一个完整例子

例如：

“2026 年 4 月选择 Clerk 替代 Auth0，因为成本和 DX 更优”

它的主路径可以是：

- `Domain = project`
- `Topic = auth`
- `Aspect = provider`
- `Kind = decision`

而事实节点再用 DSL 表达具体断言。

#### 3.1.1.7 允许值集合必须封闭，而不是开放生成

如果允许 LLM 直接创造新的 `Domain / Topic / Aspect / Kind`，系统很快就会出现：

- 同义分类并存；
- 深浅不一的路径；
- 一次性分类；
- 拼写漂移；
- 相同事实落到不同树枝。

所以这里进一步拍板：

> **主分类采用封闭词表。LLM 只能选，不能造。**

这并不意味着分类永远不扩展，而是意味着：

- 运行时是封闭集合；
- 扩展时必须人工审查；
- 新值只能通过 taxonomy 版本更新进入系统。

##### 1. Domain：taxonomy v1 最小集合

Domain 应该尽量少，只保留最大认知域。

taxonomy v1 先只保留当前项目真正高频、而且容易稳定判别的 Domain。

允许值：

| 值 | 含义 |
|---|---|
| `project` | 项目内部决策、实现、配置、里程碑 |
| `system` | 运行时、架构、存储、基础设施、模型能力 |
| `process` | 规划、执行、验证、事故、实验流程 |
| `external` | 外部供应商、依赖、上游 API、法规、市场信号 |

说明：

- Domain 不应该频繁新增。
- 如果 LLM 无法判断，优先选择最接近的现有 Domain，而不是造新 Domain。

##### 2. Topic：taxonomy v1 最小集合

Topic 不建议做成一个全局大杂烩，而应当是：

> **每个 Domain 自己有一组固定 Topic。**

这样能避免跨域同名 Topic 语义漂移。

taxonomy v1 先压缩成最小集合如下。

**`project` 的 Topic：**

- `general`
- `memory`
- `retrieval`
- `agent`
- `truth`
- `config`
- `testing`
- `docs`

**`system` 的 Topic：**

- `general`
- `architecture`
- `storage`
- `runtime`
- `model`
- `security`
- `performance`
- `integration`

**`process` 的 Topic：**

- `general`
- `planning`
- `implementation`
- `verification`
- `review`
- `experiment`

**`external` 的 Topic：**

- `general`
- `provider`
- `dependency`
- `api`
- `regulation`
- `cost`

说明：

- `general` 是唯一允许的保守兜底值。
- LLM 如果找不到精确 Topic，必须回落到 `general`，而不是发明新 Topic。
- 像 `auth`、`ingest`、`search`、`rumination` 这类更细的概念，在 taxonomy v1 中先不作为 Topic，而作为 DSL 字段、标签或图关系处理。

##### 3. Aspect：taxonomy v1 最小集合

Aspect 表示“从哪个切面看这个 Topic”。

允许值：

| 值 | 含义 |
|---|---|
| `general` | 泛化切面，无法更细分时使用 |
| `provider` | 供应方 / 实现来源 |
| `policy` | 规则、制度、策略约束 |
| `capability` | 能力边界、可做与不可做 |
| `structure` | 结构、组成、分层 |
| `interface` | 输入输出、交互面、协议面 |
| `behavior` | 运行表现、可观察行为 |
| `state` | 当前状态、健康度、模式 |
| `timeline` | 时间演化、顺序、阶段 |
| `evidence` | 证据、出处、验证材料 |
| `cost` | 成本、预算、资源消耗 |
| `risk` | 风险、脆弱点、故障可能性 |
| `constraint` | 显式限制条件 |

说明：

- Aspect 是观察角度，不是对象类型。
- `decision`、`risk` 这类词不应出现在 Aspect，而应放在 Kind。
- `performance` 在 taxonomy v1 中先并入 `behavior` 与 `state` 的判别空间，不单独立类。
- `ownership` 在 taxonomy v1 中先并入 `structure` 或 `constraint`，避免过早细分。

##### 4. Kind：taxonomy v1 最小集合

Kind 表示“这条记忆是什么类型的认知对象”。

允许值：

| 值 | 含义 |
|---|---|
| `observation` | 原始观察或尚未升格的陈述 |
| `fact` | 已相对稳定的事实性断言 |
| `decision` | 做出的选择或拍板 |
| `constraint` | 必须满足的限制 |
| `risk` | 潜在问题或失败可能性 |
| `hypothesis` | 待验证假设 |
| `pattern` | 可复用模式或规律 |
| `issue` | 当前问题、缺陷、阻塞 |
| `rule` | 规则、约定、规范 |

说明：

- Kind 是对象类型，不是主题。
- 当 LLM 不确定时，默认回落到 `observation`，而不是发明新 Kind。
- `task` 在 taxonomy v1 中先视为 `issue` 或 `constraint` 的执行外显，不单独立类。
- `preference` 在 taxonomy v1 中先并入 `rule` 或 `observation`，避免和事实/决策边界混淆。

##### 5. 运行时约束

一旦采用固定允许值集合，运行时应遵守：

1. DSL 生成器只能从白名单中选值。
2. 任一层都不允许输出未注册值。
3. 如果不确定，只能使用：
   - `Topic = general`
   - `Aspect = general`
   - `Kind = observation`
4. 白名单扩展必须通过人工审查后升级 taxonomy 版本。

也就是说，LLM 在这里不是分类发明者，而只是：

> **受限分类选择器。**

#### 3.1.1.8 分类优先用传统统计算法，而不是用 LLM 直接判类

在这套架构里，应把两件事严格分开：

- **分类**
- **摘要**

其中：

- 分类负责把事实放进稳定 taxonomy；
- 摘要负责把原文压缩成前台可调用 DSL。

这两件事虽然都处理文本，但不应由同一种机制负责。

这里进一步拍板：

> **主分类优先用传统统计算法实现，而不是用 LLM 直接输出 `Domain / Topic / Aspect / Kind`。**

原因是：

1. **分类层需要稳定，而不是灵活**
   LLM 擅长生成，但分类系统最重要的是边界稳定、同类一致、长期可比。

2. **分类层是封闭词表，更适合判别器**
   当输出集合已经固定时，问题本质上是：
   > 从有限标签中选一个最合适的标签
   
   这比生成任务更接近传统统计学习。

3. **传统统计方法更可控**
   它更容易做到：
   - 固定标签空间；
   - 概率输出；
   - 低置信度回落；
   - 可解释特征；
   - 可重复结果。

4. **LLM 最容易出问题的地方，恰好就是 taxonomy 漂移**
   它会：
   - 发明近义新类；
   - 对同类事实给出不同深度；
   - 被上下文诱导临时改分类。

而这些恰好是记忆系统最不想承受的错误。

因此更合理的分工是：

- **传统统计算法负责分类**
- **LLM 负责摘要 DSL**

##### 1. 推荐的分类实现方式

第一版不需要复杂模型，推荐用层级式传统分类器：

1. `Domain` 分类器
2. 在选定 `Domain` 后进入对应 `Topic` 分类器
3. 再进入 `Aspect` 分类器
4. 最后进入 `Kind` 分类器

也就是说，分类过程本身应是一个：

> **受 taxonomy 约束的层级判别链**

而不是一次性开放生成四层路径。

##### 2. 推荐的特征来源

可以优先使用这些传统信号：

- 词频 / TF-IDF
- 关键词词典
- n-gram
- 标题 / 路径 / 来源提示
- record_type / scope / source_kind 等结构化元数据
- 时间字段、配置字段、错误词、动作词等规则特征

也就是说，分类器不必只看自然语言正文，还应把结构化元数据一起纳入判别。

##### 3. 推荐的模型形态

第一版推荐从简单模型开始：

- 线性分类器
- Naive Bayes
- Logistic Regression
- SVM
- BM25 / TF-IDF + rule-based rerank

先做：

- 可解释
- 可控
- 可回退
- 可校正

而不是一开始追求“最聪明”。

##### 4. 低置信度处理

传统分类的一个优势，是它天然适合做低置信度回退。

如果分类器不确定，应当：

- `Topic -> general`
- `Aspect -> general`
- `Kind -> observation`

必要时打上：

- `needs_review=true`

这比让 LLM 在不确定时“猜一个看起来合理的类”更安全。

##### 5. LLM 在分类链中的位置

如果未来需要 LLM 参与，它也不应直接拥有 taxonomy 写权。

更安全的位置是：

- 只负责把原文规范化；
- 或只负责提供候选特征；
- 或只在低置信度案例中给出辅助建议；
- 但最终标签仍由封闭词表判别器和人工审查共同决定。

所以在当前设计里，更推荐的架构是：

> **分类用传统算法，摘要用 LLM，taxonomy 扩展靠人工治理。**

#### 3.1.2 事实节点应采用双层存储，而不是只有原文

事实如果只存原文，那么回忆时会太重；
事实如果只存摘要，那么证据会丢失。

因此，事实节点更合理的形式是两层存储：

1. **记忆摘要 DSL 层**
2. **原文 / 证据层**

其中：

- `记忆摘要 DSL 层` 用于前台快速调用，负责把事实压缩成可比较、可组合、可进入工作记忆的紧凑结构；
- `原文 / 证据层` 用于追溯、校验、重新解释和防止压缩失真。

这意味着“事实”不是一个字符串，而是一个小结构体：

- 它有一个高密度、低带宽的可调用表达；
- 它也有一个高保真、可审计的原始表达。

这样设计的好处是：

- 回忆时先进入前台的是压缩后的结构，而不是整段原文；
- 需要论证、复核、纠错时，系统还能回到原文；
- 摘要层负责认知效率，原文层负责认识论约束。

这里的摘要层不再预设采用固定外部速记方言，而是采用：

> **由 LLM 生成的、自定义的、面向记忆事实的 DSL。**

原因是：

- 规则压缩器更适合做索引信号提取，不一定适合表达复杂事实；
- LLM 更擅长把原文重写成规范化断言；
- 只要 DSL 的槽位稳定、语义清晰，LLM 既能写，也能读，还能继续在前台比较和组合。

因此，推荐流程是：

1. 原文进入记忆系统；
2. LLM 读取原文并生成一条或多条事实 DSL；
3. DSL 作为前台摘要层存储；
4. 原文继续作为证据层保留。

#### 3.1.3 记忆摘要 DSL 应该长什么样

一个适合 LLM 的记忆摘要 DSL，不应追求“人类最短”，而应追求：

- **槽位固定**
- **语义单义**
- **一眼可解析**
- **适合逐行扫描**
- **对 token 成本友好**

因此，推荐采用“单事实单行、短键名、固定字段顺序”的 DSL。

这里继续拍板：

> **DSL v1 采用“固定语法骨架 + 必填核心字段 + 受控选填字段”。**

也就是说：

- 不走“每条都必须硬填 8 个以上字段”的路线；
- 不走“字段完全自由、想写什么就写什么”的路线；
- 而是采用少量硬骨架，再在尾部附加少量受控扩展字段。

例如：

```text
F|DOM=project|TOP=retrieval|ASP=behavior|KIND=decision|CLAIM=use lexical-first as baseline|TRUTH=T2|SRC=roadmap#phase9|WHY=explainability and stability|TIME=2026-04
```

这条 DSL 的设计要点是：

- `F` 表示这是一条事实；
- `DOM` 表示 Domain；
- `TOP` 表示 Topic；
- `ASP` 表示 Aspect；
- `KIND` 表示 Kind；
- `CLAIM` 表示规范化断言本身；
- `TRUTH` 表示它位于 T1/T2/T3 的哪一层；
- `SRC` 指向原文证据锚点。

这样的 DSL 比自由摘要更适合记忆系统，因为它让 LLM 看到的不是一段“写得漂亮的话”，而是一组位置稳定、功能稳定的认知槽位。

进一步说，这个 DSL 的目标不是文学表达，而是：

- 让系统能快速对齐两个事实是否相同；
- 让工作记忆能快速装配事实；
- 让元认知能检查“这条断言有没有来源、时间、真值层”；
- 让后续检索和比较可以面向字段，而不是面向整段文本。

#### 3.1.3.1 为什么不做“全字段强制必填”

事实不是同质对象。

- 决策类事实通常天然带 `WHY`
- 风险类事实更天然带 `IMPACT`
- 观察类事实可能只有 `CLAIM + SRC + TRUTH`

如果强迫所有事实都填满同一批字段，系统会出现：

- 大量占位符；
- LLM 为了补齐字段而编造内容；
- 行长度膨胀，压缩效果变差；
- 真正重要的字段被格式噪音淹没。

因此，固定骨架应该尽量少，只保留最小可成立的事实表达。

#### 3.1.3.2 为什么也不做完全自由字段

如果每条事实想写什么字段就写什么字段，那么：

- 解析器会变复杂；
- 工作记忆装配会失去稳定模式；
- LLM 对字段顺序的预期会被破坏；
- 同类事实很难做批量比较。

所以更稳的做法是：

> **必填字段少而硬，选填字段少而受控。**

#### 3.1.3.3 DSL v1 字段规范

DSL v1 规定：

- 行前缀固定为 `F|`
- 字段顺序固定
- 必填字段必须全部出现
- 选填字段如果缺失，则直接省略
- 不允许使用空占位符，如 `WHY=-` 或 `TIME=unknown`

##### 必填核心字段

以下 7 个字段必须始终存在：

1. `DOM`
2. `TOP`
3. `ASP`
4. `KIND`
5. `CLAIM`
6. `TRUTH`
7. `SRC`

它们分别负责：

- `DOM`：最大认知域
- `TOP`：域内稳定主题
- `ASP`：观察切面
- `KIND`：认知对象类型
- `CLAIM`：规范化断言本身
- `TRUTH`：T1 / T2 / T3 真值层
- `SRC`：证据锚点

如果没有这 7 个字段，就不能称为一条完整事实。

##### 受控选填字段

DSL v1 允许以下 6 个扩展字段：

1. `WHY`
2. `TIME`
3. `COND`
4. `IMPACT`
5. `CONF`
6. `REL`

它们的职责分别是：

- `WHY`：原因、依据、动机
- `TIME`：发生时间、有效时间、版本时间
- `COND`：成立条件、适用前提、边界条件
- `IMPACT`：后果、影响范围、作用结果
- `CONF`：主观置信度或结构置信度
- `REL`：与其他事实的关系引用

这些字段不是自由扩展区，而是：

> **白名单中的受控尾字段。**

也就是说，DSL v1 不允许在运行时临时发明例如 `MOOD=`、`STYLE=`、`COMMENT=` 这样的新键。

#### 3.1.3.4 字段顺序规则

DSL v1 的固定顺序为：

```text
F|DOM=...|TOP=...|ASP=...|KIND=...|CLAIM=...|TRUTH=...|SRC=...|WHY=...|TIME=...|COND=...|IMPACT=...|CONF=...|REL=...
```

规则如下：

1. 前 7 个必填字段必须按顺序出现。
2. 选填字段如果出现，也必须按上述顺序追加。
3. 不允许调换顺序。
4. 不允许在中间插入未知字段。

这样做的好处是：

- LLM 读写时形成稳定节奏；
- parser 可以线性解析；
- 下游模块可以固定槽位消费。

#### 3.1.3.5 选填字段何时使用

为了防止行长度失控，选填字段应遵循最小必要原则：

- 只有缺少该字段会显著损害理解时才写；
- 能由 `CLAIM` 直接承载的内容，不重复写到 `WHY`；
- 只有存在明确时间约束时才写 `TIME`；
- 只有存在明确条件边界时才写 `COND`；
- 只有该事实天然具有结果面向时才写 `IMPACT`；
- `CONF` 和 `REL` 应谨慎使用，不作为默认字段。

换句话说，DSL v1 追求的是：

> **一条短而完整的事实，而不是一条填满表格的事实。**

#### 3.1.3.6 与 AAAK 的类比

如果做一个类比：

- AAAK 更像“面向 agent 的压缩速记协议”
- 这里的 DSL 更像“面向记忆事实的规范断言协议”

AAAK 的优点是全行协议稳定、可快速扫读；
而这里的 DSL 保留了这种稳定性，但更强调：

- 事实本体槽位；
- 真值层与证据锚点；
- 长期存储与工作记忆装配兼容；
- 运行时治理，而不是仅仅输出好看。

因此，DSL v1 的目标不是“像速记一样酷”，而是：

> **让事实可以被稳定生成、稳定解析、稳定比较、稳定治理。**

#### 3.1.3.7 Kind 到字段的推荐映射

虽然 DSL v1 的字段集合已经固定，但并不意味着每种 `Kind` 都应使用同样的字段组合。

更合理的做法是：

> **字段集合统一，字段使用习惯按 Kind 收敛。**

也就是说，`Kind` 决定“这条事实最自然应该长成什么样”。

##### 推荐映射表

| Kind | 必填字段 | 推荐选填字段 | 谨慎使用字段 | 说明 |
|---|---|---|---|---|
| `observation` | `DOM/TOP/ASP/KIND/CLAIM/TRUTH/SRC` | `TIME` | `WHY`, `IMPACT`, `CONF`, `REL` | 观察类事实优先保留原始观察，不应过度解释。 |
| `fact` | `DOM/TOP/ASP/KIND/CLAIM/TRUTH/SRC` | `TIME`, `COND` | `WHY`, `IMPACT`, `CONF`, `REL` | 事实强调成立条件与时间有效性，不强调主观理由。 |
| `decision` | `DOM/TOP/ASP/KIND/CLAIM/TRUTH/SRC` | `WHY`, `TIME`, `IMPACT` | `CONF`, `REL` | 决策如果没有 `WHY`，通常会损失可解释性。 |
| `constraint` | `DOM/TOP/ASP/KIND/CLAIM/TRUTH/SRC` | `COND`, `TIME` | `WHY`, `IMPACT`, `CONF`, `REL` | 约束类对象重点是边界和适用条件。 |
| `risk` | `DOM/TOP/ASP/KIND/CLAIM/TRUTH/SRC` | `IMPACT`, `COND`, `TIME` | `WHY`, `CONF`, `REL` | 风险类对象最重要的是后果和触发条件。 |
| `hypothesis` | `DOM/TOP/ASP/KIND/CLAIM/TRUTH/SRC` | `COND`, `CONF`, `TIME` | `IMPACT`, `REL` | 假设类对象应明确不确定性，不应伪装成稳定事实。 |
| `pattern` | `DOM/TOP/ASP/KIND/CLAIM/TRUTH/SRC` | `WHY`, `COND`, `REL` | `CONF` | 模式类对象通常需要说明适用边界与支持关系。 |
| `issue` | `DOM/TOP/ASP/KIND/CLAIM/TRUTH/SRC` | `IMPACT`, `TIME`, `COND` | `WHY`, `CONF`, `REL` | 问题类对象最重要的是影响与触发上下文。 |
| `rule` | `DOM/TOP/ASP/KIND/CLAIM/TRUTH/SRC` | `WHY`, `COND`, `TIME` | `IMPACT`, `CONF`, `REL` | 规则类对象通常要说明设立理由和适用范围。 |

##### 使用原则

1. **推荐字段不是强制字段**
   它表示“这类事实最自然应包含哪些扩展信息”，而不是“必须把所有推荐字段填满”。

2. **一类事实优先只写 1-3 个选填字段**
   选填字段过多会让 DSL 退化成表单，而不是摘要。

3. **同一个 Kind 应保持风格一致**
   例如：
   - `decision` 里经常写 `WHY`
   - `risk` 里经常写 `IMPACT`
   - `hypothesis` 里经常写 `CONF`

4. **不要把字段职责互相挤占**
   例如：
   - 不能把理由塞进 `CLAIM`
   - 不能把影响塞进 `WHY`
   - 不能把不确定性藏在措辞里而不写 `CONF`

#### 3.1.3.8 Kind 的字段使用禁忌

为了避免 DSL 漂移，每种 `Kind` 还应有明确禁忌。

##### `observation`

- 不应默认补 `WHY`
- 不应轻易写 `CONF`

因为 observation 的职责是“先把看到的东西记下来”，而不是解释它。

##### `fact`

- 不应轻易写主观置信度
- 不应把“推测原因”包装成事实理由

因为 fact 要尽量保持陈述稳定。

##### `decision`

- 不应缺失 `WHY`，除非决策原因确实未知

因为没有理由的 decision 很难被未来主体正确复用。

##### `risk`

- 不应只写风险对象，不写后果

因为没有 `IMPACT` 的 risk 会退化成模糊担忧。

##### `hypothesis`

- 不应缺失 `CONF`

因为 hypothesis 本质上就是带不确定性的结构。

##### `rule`

- 不应只写命令，不写边界

因为没有 `COND` 的 rule 很容易在错误上下文中被误用。

#### 3.1.3.9 Kind 的标准例子

下面给出 DSL v1 的标准例子，用于约束生成风格。

##### `observation`

```text
F|DOM=system|TOP=runtime|ASP=state|KIND=observation|CLAIM=doctor reports embedding channel gated|TRUTH=T2|SRC=status#doctor-output|TIME=2026-04-17
```

##### `fact`

```text
F|DOM=project|TOP=retrieval|ASP=structure|KIND=fact|CLAIM=ordinary retrieval keeps lexical-first as baseline|TRUTH=T2|SRC=roadmap#phase9|TIME=2026-04
```

##### `decision`

```text
F|DOM=project|TOP=retrieval|ASP=behavior|KIND=decision|CLAIM=use lexical-first as baseline|TRUTH=T2|SRC=roadmap#phase9|WHY=preserve explainability and stable recall|TIME=2026-04
```

##### `constraint`

```text
F|DOM=system|TOP=runtime|ASP=constraint|KIND=constraint|CLAIM=embedding cannot replace lexical baseline in v1|TRUTH=T1|SRC=project#constraints|COND=all ordinary retrieval flows
```

##### `risk`

```text
F|DOM=system|TOP=model|ASP=risk|KIND=risk|CLAIM=llm-generated taxonomy can drift over time|TRUTH=T2|SRC=design-review#taxonomy|IMPACT=inconsistent classification and unstable retrieval|COND=if classification is delegated to free-form generation
```

##### `hypothesis`

```text
F|DOM=process|TOP=experiment|ASP=behavior|KIND=hypothesis|CLAIM=tfidf plus rule rerank is enough for taxonomy v1|TRUTH=T3|SRC=notes#classifier-plan|CONF=0.62|COND=current project corpus remains small
```

##### `pattern`

```text
F|DOM=project|TOP=agent|ASP=structure|KIND=pattern|CLAIM=follow-up evidence should re-enter working memory before decision selection|TRUTH=T2|SRC=phase7-summary|WHY=prevents retrieval/report divergence|REL=phase7-follow-up-integration
```

##### `issue`

```text
F|DOM=process|TOP=verification|ASP=behavior|KIND=issue|CLAIM=milestone close can drift from raw state files|TRUTH=T2|SRC=audit#v1.1|IMPACT=workflow routing may mis-detect next step|TIME=2026-04-17
```

##### `rule`

```text
F|DOM=project|TOP=docs|ASP=policy|KIND=rule|CLAIM=taxonomy values must come from a closed whitelist|TRUTH=T1|SRC=architecture-doc#taxonomy-v1|WHY=prevent llm-invented category drift|COND=all runtime classification paths
```

因此，真正的“记住”不是把原文塞进库里，
而是把原文锚定为证据，再把它压缩成未来可调用的事实表示。

与共享事实、私有假设和受控晋升的关系，可继续见 [0415-真值层.md](./0415-真值层.md)。

### 3.2 世界模型

世界模型负责表示“世界是什么样”，并且内置预测能力。

它不仅回答：

- 现在世界是什么样；
- 过去是怎么变来的；

还回答：

- 如果采取动作 A，世界可能会怎么变。

因此，世界模型不是静态状态表，而是带动力学的模型。

详细展开见：

- [0415-世界模型.md](./0415-世界模型.md)

### 3.3 自我模型

自我模型不是“人格包装”，而是主体控制模型。

它优先包含：

- 能力边界
- 当前目标
- 当前资源状态
- 当前风险状态
- 近期失败签名

自我模型的更新，主要不是靠叙述，而是靠行动结果。

详细展开见：

- [0415-自我模型.md](./0415-自我模型.md)

### 3.4 技能记忆层

技能记忆层保存的是程序性知识。

它既不是世界事实，也不是一次性计划，而是跨任务可复用的程序模板。

例如：

- 遇到不确定时先澄清；
- 先查 trace 再查 object；
- 从成功轨迹中提炼出的动作模式。

详细展开见：

- [0415-技能记忆.md](./0415-技能记忆.md)

### 3.5 注意力系统

注意力系统决定哪些后台结构进入前台。

它不是简单过滤器，而是一个动态调度器：

- 情境决定相关性；
- 自我模型决定可用性；
- 元认知决定这次应如何思考。

### 3.6 工作记忆

工作记忆不是缓存，而是临时控制场。

它不仅容纳当前状态，还容纳近端行动分支。
因此它不是被动镜像，而是主动决策场。

### 3.7 行动系统

行动系统负责：

- 策略
- 计划
- 执行

其中：

- 预测属于世界模型
- 策略与计划属于行动系统

### 3.8 元认知层

元认知层不是提示器，而是监督器。

它负责：

- 监控不确定性
- 触发澄清
- 检查风险
- 暂停错误推进
- 必要时拥有 veto 权

详细展开见：

- [0415-元认知层.md](./0415-元认知层.md)

### 3.9 价值层

价值层回答：

> 什么样的行动算好行动？

它由两部分构成：

- 固定规则基底
- 可学习调节层

固定规则保证底线，可学习参数提供情境下的取舍弹性。

## 4. T 层：真值结构，而不是摘要层

必须把两种“层”分开：

- `L层`：内容披露层（如 L0/L1/L2）
- `T层`：真值结构层（Truth Layers）

本文中采用 T 层来描述共享现实与个体认知之间的层级关系。

### 4.1 T1：共享本体层

T1 不是日常事实表，而是世界骨架。

它回答：

- 世界中有哪些稳定对象类型？
- 有哪些长期不轻易变化的关系结构？
- 哪些约束已足够稳定，值得成为本体层？

T1 是慢变量。
它通常不会频繁变化，且一旦显式化，就会影响整个系统的认知组织方式。

### 4.2 T2：共享演化 / 证据层

T2 是共享现实流。

它包含：

- 公共证据
- 时间演化
- 被验证过的共享结构
- 可追溯的共享事实

它不是最终本体，但它是公共世界的活层。

### 4.3 T3：私有认知映射层

T3 是每个 agent 的私有工作假设层。

它包含：

- 私人解释
- 经验偏置
- 局部映射
- 风险偏向
- 临时工作假设

T3 允许暂时与共享层不一致，但必须：

- 显式标记
- 可回溯
- 可撤销
- 不得直接改写 T1/T2

### 4.4 T 层不是静态分类，而是演化结构

T 层有两条关键跃迁：

- T3 → T2：从个体认知到共享演化
- T2 → T1：从共享演化到共享本体

这两条跃迁都必须受控进行，而不是自然漂移。

T 层详细展开见：

- [0415-真值层.md](./0415-真值层.md)

## 5. 注意力系统：什么先进入前台

注意力系统是认知前台的发动机。

它的核心不是“查什么”，而是：

> **在当前处境下，什么值得先被想起。**

### 5.1 注意力的主要驱动力

在本文框架中，注意力主要由：

- 情境 / 身体状态

驱动，而不是由目标单独驱动。

目标不消失，但它以**偏置层**的形式存在，而不是作为唯一发动机。

### 5.2 情绪的地位

情绪不只是情境向量中的一个维度。
它更高一层：

> **情绪是匹配函数的调节器。**

也就是说，情绪改变的不是事实本体，而是：

- 匹配宽度
- 激活阈值
- 抑制强度
- 偏好方向

### 5.3 状态输入的边界

状态输入只能改变：

- 召回优先级

不能直接改变：

- 真值
- 有效性
- 证据约束

因此必须严格区分：

- 世界模型真值通道
- 注意力优先级通道

### 5.4 注意力的双时间尺度

注意力状态不是单层，而是双层：

$$
AttentionState(t) = Baseline(t) + Delta(task_t, metacog_t)
$$

其中：

- `Baseline` 是会话级持续状态
- `Delta` 是回忆前的临时偏置

详细定义见：

- [0415-注意力状态.md](./0415-注意力状态.md)

---

## 6. 工作记忆：当前帧与近端分支

工作记忆不是简单的前台缓存。

它是：

> **注意力输出装配成的临时控制场。**

### 6.1 工作记忆的本质

工作记忆里不仅有“现在是什么”，还有“下一步可以是什么”。

因此它是：

- 当前帧
- 近端分支

的复合体。

### 6.2 工作记忆中的近端分支

这些分支不是抽象口号，也不是完整计划树，而是：

> **已握在手中、参数已填充、后果已预览、价值已标价的半结构化行动候选。**

### 6.3 三类行动的共同竞技场

工作记忆中的 `candidate_actions` 不只包含操作性行动。
它同时包含：

- 认识性行动
- 操作性行动
- 调控性行动

因此工作记忆不是执行前厅，而是：

> **认知主体权衡轻重、审时度势的真正决策场。**

工作记忆的完整形式化见：

- [0415-工作记忆.md](./0415-工作记忆.md)

## 7. 价值层：什么算好行动

如果工作记忆中有三类行动共同竞争，那么系统必须回答一个问题：

> 它们如何在同一个价值尺度下被比较？

### 7.1 价值层的双层结构

价值层由两部分组成：

- 固定规则基底
- 可学习调节参数

固定规则保证不可突破的底线。  
可学习参数决定在允许空间中的偏向。

### 7.2 多维价值，而不是单一分数

价值层不应直接输出一个固定标量。
它首先输出多维价值向量，例如：

- 目标推进度
- 信息增益
- 风险规避
- 资源效率
- 主体稳健性

然后在当前情境、自我状态和元认知状态下，
把多维价值向量投影为统一标量。

因此：

> **价值层不是静态评分器，而是动态价值投影器。**

### 7.3 元认知的 veto 权

即使某行动在价值上分数最高，元认知层仍然可以拥有独立 veto 权。

这意味着：

- 价值层负责评价
- 元认知层负责监督

二者不是替代关系，而是互补关系。

价值层详见：

- [0415-价值层.md](./0415-价值层.md)

## 8. 反刍：从被动记录到主动学习

如果没有反刍，认知系统只能运行，不能成长。

反刍的定义不是“后台整理”，而是：

> **认知系统对经验进行主动重组织、重估值和重写回各层模型的学习过程。**

### 8.1 反刍的两种时间尺度

反刍不是一个单一过程，而是两种节律：

- 短周期反刍：即时止血
- 长周期反刍：缓慢消化

短周期反刍优先改写：

- 自我模型
- 风险敏感度
- 局部认知映射

长周期反刍优先改写：

- 世界模型
- 技能记忆层
- 价值层慢变量
- T层晋升候选

### 8.2 反刍的双队列

反刍必须双队列分离：

- SPQ：短周期同步队列
- LPQ：长周期异步队列

这不是实现优化，而是架构地基。

### 8.3 反刍写回的性质

反刍不是空转推理，它必须写回系统长期状态。

因此反刍模块本质上是：

> **主动学习控制面 / learning control plane**

反刍的详细形式化见：

- [0415-反刍机制.md](./0415-反刍机制.md)

## 9. 认知系统的主循环

把前面的层全部连起来，可以得到这套认知系统的核心循环：

1. 世界和主体产生新经验
2. 记忆系统把经验保存为可调用结构
3. 世界模型维护“世界是什么样”
4. 自我模型维护“我当前如何面对这个世界”
5. 注意力系统决定哪些结构进入前台
6. 工作记忆把当前帧与近端分支装配为控制场
7. 价值层为候选行动赋予多维价值
8. 元认知层进行最终监督，必要时 veto
9. 行动系统执行
10. 行动结果反向更新世界模型、自我模型与反刍系统

如果把它写成一句总纲：

> **后台世界与主体模型通过注意力被带入前台，  
> 在工作记忆中形成当前帧与近端分支，  
> 由价值层进行多维评估，  
> 由元认知进行最终监督，  
> 再由行动系统执行并把结果反馈回后台。**

## 10. 术语与符号表

### 10.1 核心术语

- **记忆**：服务未来行动的结构化保存、持续更新、可解释调用机制；它既包括后台长期结构，也包括这些结构如何重新进入前台。
- **结构化记忆树**：按“主题 → 分类 → 事实”等抽象层级组织起来的记忆结构。它不是展示目录，而是认知组织本身。
- **主路径**：一条事实在结构化记忆树中的唯一挂载位置。本文推荐采用 `Domain / Topic / Aspect / Kind / Fact`。
- **Domain**：最大认知域，负责决定这条记忆首先属于哪个世界区域。
- **Topic**：域内稳定主题，负责确定记忆主要在谈哪块对象。
- **Aspect**：主题的观察切面，负责标出这条记忆是在谈主题的哪个方面。
- **Kind**：认知对象类型，负责区分 decision / fact / risk / constraint / hypothesis / pattern 等不同性质的内容。
- **经验材料**：世界或主体经历过的原始内容，如事件、观察、日志、会话片段、结果记录。经验材料不是记忆本体，而是记忆原料。
- **事实节点**：结构化记忆树的叶节点。它不是孤立原文，而是一个带位置、带证据、可被追溯和修订的事实单元。
- **记忆摘要 DSL**：事实节点的高密度可调用表达，由 LLM 根据原文生成，用于前台快速回忆、比较和装配；它与原文层并存，而不替代原文。
- **LLM 摘要器**：把原文转换为记忆摘要 DSL 的生成模块。它的职责不是“写得好看”，而是把原文压缩成槽位稳定的事实表达。
- **图关系**：事实之间的附加连接层，用来表达 `relates_to`、`depends_on`、`contradicts` 等跨主路径联系。它不替代树的主路径。
- **纯检索系统**：原生支持状态输入的 recall engine。它负责在不改写真值的前提下，按当前状态召回最值得进入前台的结构。
- **认知系统**：在纯检索系统之上，加入自我模型、工作记忆、行动、元认知与价值层的主体系统。
- **世界模型**：表示世界并内置预测能力的模型。它负责“世界是什么样”和“如果这样做，世界会怎样变”。
- **自我模型**：表示主体当前能力、资源、风险、负载和边界的控制模型。它回答“我现在如何面对这个世界”。
- **技能记忆**：由成功轨迹压缩出来的程序性知识，位于策略与单次计划之间，为行动系统提供可复用模板。
- **注意力**：决定什么进入前台的调度机制。它不是单纯查询，而是对“什么值得先被想起”的状态化计算。
- **工作记忆**：当前帧与近端分支的临时控制场。它不只是内容集合，还包含行动可供性。
- **行动系统**：负责策略、计划、执行与动作选择的层。预测不属于它，但行动选择属于它。
- **元认知**：监督认知过程本身的层。它负责监控不确定性、调节 recall policy，并在必要时拥有 veto 权。
- **价值层**：决定“什么算好行动”的评估层。它先输出多维价值向量，再在当前权重配置下投影成标量。
- **反刍**：主动学习控制面。它读取经验、结果与前台残留，跨层写回世界模型、自我模型、技能层、价值层及 T 层候选。

### 10.2 T 层术语

- **T层**：Truth Layers，专门描述共享真值、证据与私有认知映射的层级结构；与内容展示层 `L层` 严格区分。
- **T1**：共享本体层，表示世界骨架、长期稳定结构与认知系统的本体约束，是慢变量。
- **T2**：共享演化 / 证据层，表示共享现实流、时态事实和可追溯证据，是共享世界的活层。
- **T3**：私有认知映射层，表示个体主体的工作假设、局部解释、风险偏置和任务投影，是快变量。

### 10.3 前台术语

- **AttentionState**：注意力状态总对象，本文采用双时间尺度结构，由 `Baseline` 与 `Delta` 叠加形成。
- **Baseline**：会话级持续状态，是情境底色、情绪底色和长期目标方向的缓慢演化背景。
- **Delta**：回忆前临时偏置，由当前任务、瞬时情绪波动与元认知警报触发，检索后即销毁。
- **ContextBase**：情境基底向量，是 AttentionState 的主环境部分，表达当前处境本身。
- **EmotionModulator**：情绪匹配调节器，不直接定义内容，而改变匹配函数的增益、阈值和抑制模式。
- **GoalBias**：目标偏置层，不作为注意力主引擎，而作为情境之上的方向性偏置。
- **world_fragments**：由注意力系统从长期结构中筛出的当前世界片段，进入工作记忆的情境场。
- **self_state**：自我模型的前台切片，是某一时刻主体运行状态的快照，而不是完整自我模型。
- **candidate_actions**：工作记忆中的半结构化行动候选，已具备参数、预期后果、风险与价值评分。
- **metacog_flags**：元认知层注入前台的控制标志，例如暂停、降级、强制澄清等。

### 10.4 行动与学习术语

- **认识性行动**：以减少不确定性为目标的行动，如查证据、问澄清、看 timeline。
- **操作性行动**：以直接改变世界状态为目标的行动，如执行工具、修改配置、发送请求。
- **调控性行动**：以改变主体自身认知状态为目标的行动，如暂停、降级、切保守模式、请求人工介入。
- **多维价值向量**：行动在多个异质维度上的价值表示，如目标推进、信息增益、安全性、效率、稳健性。
- **价值投影**：将多维价值向量在当前权重配置下压缩成单一标量，用于统一比较行动候选。
- **SPQ**：短周期反刍同步队列，面向止血、即时纠偏和局部运行时写回。
- **LPQ**：长周期反刍异步队列，面向巩固、抽象、技能形成、共享结构固化与价值慢学习。
- **T3→T2 晋升**：从个体认知映射到共享现实层的受控固化。
- **T2→T1 显式化**：从共享演化层到共享本体层的受控本体化。

### 10.5 主要公式索引

- 注意力双层状态：

$$
AttentionState(t) = Baseline(t) + Delta(task_t, metacog_t)
$$

- 工作记忆总定义：

$$
WorkingMemory(t)=\{world\_fragments, self\_state, active\_goal, active\_risks, candidate\_actions, metacog\_flags\}
$$

更详细的注意力、工作记忆、价值层与反刍公式，见对应专题文档。

---

## 11. 结语：记忆不是过去的保存器，而是未来的准备器

如果要把全文压缩成一句话，那就是：

> **记忆不是过去的保存器，而是认知系统在价值约束下为未来行动准备世界模型的机制。**

这句话里包含了本文的全部核心判断：

- 记忆服务未来，而不是服务归档；
- 记忆的本体是可调用结构，而不是原始文本；
- 记忆通过世界模型进入认知，通过注意力进入前台，通过工作记忆形成控制场，通过行动系统走向现实；
- 反刍让系统不只是记录经验，而是主动学习；
- T 层让共享真值与个体认知能够同时存在而不互相污染。

因此，真正的记忆系统从来不是孤立模块。
它始终是认知系统的一部分。
